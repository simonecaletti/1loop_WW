# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

This is not software in the conventional sense — it is a **FORM** (symbolic
manipulation program for high-energy physics, https://www.nikhef.nl/~form/)
computation for the one-loop QCD amplitude `gg -> V V` (two gauge/vector
bosons produced via a gluon-gluon initiated quark loop, e.g. `Vboson1`,
`Vboson2` could be `W`/`Z`), with diagrams generated by **qgraf**. There is no
build system, package manager, or test suite; "running" this project means
invoking the FORM interpreter (`form` / `tform`) on the `.frm` files in
sequence, which read/write the `.inc` files as an on-disk cache/pipeline.

There are no commands to build, lint, or test — correctness is checked by
running each `prog_N_*.frm` stage through FORM and inspecting the printed
output / `.inc` files it writes.

- **Mathematica** — available as `math` / `wolframscript` / `MathKernel` on

  evaluate the master integrals into
  `../Numerical_Evaluation_1loop/AMFlow_results{,_large_mt}/`; not directly

## Running the pipeline

Run each stage from inside `1loop_WW/` (the `#include` paths in the `.frm`
files are relative to this directory), strictly in numeric order — each
stage reads `.inc` files the previous stage wrote:

```
form prog_1_ggVV_1L.frm          # writes Diagrams/diagram<N>.inc (N=1..8)
form prog_2_ggVV_1L.frm          # reads Diagrams/, writes Reduze-notation output
# --- external Reduze IBP reduction happens here, not runnable in this repo ---
form prog_3_ggVV_1L.frm          # reads Tensors_x_amp/AmpT<N>d.inc (N=1..36)
form prog_4_ggVVFF_1L.frm        # writes Form_Factors/FF<N>.inc (N=1..36)
form prog_4_ggVVLL_1L.frm        # writes Helicity_form_factors/He<N>LL.inc (N=1..18)
form prog_4_ggVVLR_1L.frm        # writes Helicity_form_factors/He<N>LR.inc (N=1..18)
```

Use `tform -w<N>` instead of `form` for the heavier stages (`prog_3`,
`prog_4`) if wall-clock time matters — the `.frm` files don't hard-code a
worker count. A stage is considered to have run correctly if FORM exits
without an error and the `.inc` files it names in its trailing `#write`
block are (re)written; there's no separate check beyond inspecting that
output. The `prog_4` variants (`FF`/`LL`/`LR`) are independent of each other
and can be run in any order once `prog_3` has produced
`Tensors_x_amp/AmpT<N>d.inc`.

## This directory is one component of a larger project

Several `#include` paths in the `.frm` files point one level up, to sibling
directories that live outside `1loop_WW/` (i.e. in its parent directory, not
inside this repo/working directory):

- `../Procedures/Procedures.proc`, `../Procedures/Procedures_2.proc` — shared
  FORM `#Procedure`s used by every `prog_N` stage: `kinematics`/
  `allsimplnosort` (Mandelstam substitutions `p1.p2=s/2` etc. and momentum
  conservation `p4=-p1-p2-p3`), `transversality`/`gaugechoice` (polarization
  vector transversality conditions), `coloralgebra`/`fiertz` (SU(N) color
  algebra: `f`-to-`T` rewriting, Fierz identity, trace building), `Levi_Civita`
  (`e_` contraction identities), `FactArgNum`/`FactArgDen` (factor `Num(...)`/
  `Den(...)` arguments), `namemomenta` (propagator momentum → symbolic label
  renaming, e.g. `k1+p1 -> k1p1`, used before matching diagrams to topologies).
- `../Styles/form.sty.mine`, `../Styles/reduze.sty` — qgraf output-formatting
  templates (not FORM/Reduze code themselves) that control how qgraf renders
  `ggVV.1loop` / `ggVV.1loop.yaml` from its internal diagram representation.
- `../Models/weak_qcd_form_WW`, `weak_qcd_reduze_WW`, `weak_qcd_form_ZZ`,
  `weak_qcd_reduze_ZZ` — qgraf model files declaring particle content
  (`Quark1`/`Qbar1` massless, `Quark2`/`Qbar2` mass `mt`, `gluon`, `ghost`,
  `Vboson1`/`Vboson2` with masses `m3`/`m4`) and allowed vertices, including
  the flavour-changing `[Qbar2,Vboson1,Quark1]` / `[Qbar1,Vboson2,Quark2]`
  vertices that let a light/heavy quark pair produce the two bosons. The
  `_WW` vs `_ZZ` and `_form` vs `_reduze` variants pick the process and the
  qgraf output style respectively; `1loop_WW` here uses `_form_WW` (stage 1)
  and `_reduze_WW` (the `.yaml` dump feeding topology/reduction work).
- `../Tensor_decomposition/` — where the tensor basis and projectors used by
  `prog_3`/`prog_4` are derived (largely outside FORM, in Mathematica):
  `tensors.inc` defines the 36 tensor structures `T1..T36` (contractions of
  `eps1..eps4`, `p1..p3`, `va`); `projectorscoeffs.inc` is the (huge,
  generated) set of `c<i>p<j>` projector coefficients `prog_4_ggVVFF` sums
  against `AmpT<j>d`; `helicity_coeff_LL.inc` / `helicity_coeff_LR.inc` hold
  the `FtoHe(<i>,<j>)` form-factor-to-helicity-amplitude coefficients used by
  `prog_4_ggVVLL`/`LR`; `Prog_tensors_ggVV.frm` and the `.nb`/`.m` files
  (`Spinor_chains_ggVV.nb`, `TensorMetric_ggVV.m`, `TensorMetric.m`) are the
  Mathematica/FORM derivation behind those `.inc` files — treat the `.inc`
  files as the generated artifacts actually consumed by the pipeline.
- `../Numerical_Evaluation_1loop/` — downstream of this directory, not
  upstream: `prog_4_ggVV{FF,LL,LR}_1L.frm` write their final `.m` results into
  `gg1l_results_AG/` (`FF<N>.m`, `He<N>_LL.m`, `He<N>_LR.m`) for numerical
  evaluation in Mathematica notebooks here (`Numerical_evaluation_1loop.nb`,
  `ggVV_1l_LS.nb`). `AMFlow_results/` and `AMFlow_results_large_mt/` hold
  independently-computed master-integral values (via AMFlow) per topology
  label (`PL00M`, `PL0M0`, `NPL0M`, and `x12` partners) used as numerical
  cross-checks/inputs; `gg1l_results/` holds derived helicity-amplitude
  results (`ggVV_LL.m`, `ggWW_helamp_exact.m`, etc.); `cache/` is scratch
  output from AMFlow runs.

`qgraf.dat` is a binary file (qgraf configuration/output), not text.

## Pipeline / data flow

The computation is a strict multi-stage pipeline. Each stage is one
`prog_N_*.frm` file, executed in order. FORM reads symbols declared at the
top, includes shared procedure files, does algebra, and writes results into
the `.inc` directories, which the next stage reads back in.

1. **qgraf output** — `ggVV.1loop` (FORM-style diagram list, `id d1=...`
   `id d2=...` etc., one per Feynman diagram) and `ggVV.1loop.yaml`
   (human-readable YAML dump of the same diagrams: external legs,
   propagators, vertices) are both generated by qgraf from `qgraf.dat` and
   are treated as inputs, not hand-edited.

2. **`prog_1_ggVV_1L.frm`** — Imports the 8 diagrams from `ggVV.1loop`,
   applies momentum conservation (`p4 = -p1-p2-p3`), substitutes Feynman
   rules for propagators/vertices/polarizations, builds closed fermion loops
   (`LoopF`), does color algebra (traces of `T`/`f`), contracts with
   polarization vectors, and applies kinematics/transversality/gauge-choice
   procedures. Writes one file per diagram to `Diagrams/diagram<N>.inc`
   (`N` = 1..8) via `#write`.

3. **`prog_2_ggVV_1L.frm`** — Reads `Diagrams/diagram<N>.inc` back in (via
   `ggVV.1loop`/prog_1 diagram labels), applies conjugate tensors to the
   amplitude, and substitutes Reduze-style notation needed for master
   integral reduction (propagator labels `D1..D15`, sector labels like
   `PL00M`, `PL0M0`, `NPL0M` defined via `Topos1L.rec`).

4. **External: integral reduction** happens outside this directory (Reduze
   or similar, feeding off `Topos1L.rec` sector definitions), and master
   integrals are evaluated numerically via AMFlow into
   `../Numerical_Evaluation_1loop/AMFlow_results{,_large_mt}/` — not visible
   here.

5. **`prog_3_ggVV_1L.frm`** — Includes `Tensors_x_amp/AmpT<N>d.inc` (N=1..36,
   tensor structures contracted with the reduced amplitude, each a big FORM
   expression in `INT(...)`, `Num(...)`, kinematic invariants `s`, `t`, `u`,
   `m32`, `m42`, `mt`). This is where form-factor decomposition coefficients
   (`c<i>p<j>`, defined in `../Tensor_decomposition/projectorscoeffs.inc`)
   would combine `AmpT<N>d` terms into 36 form factors `F<i>` (that
   combination step is commented out in `prog_3`, done instead in `prog_4`).

6. **`prog_4_ggVV{FF,LL,LR}_1L.frm`** — Three sibling variants of the final
   stage, differing only in what basis the result is projected onto:
   - `FF` — builds the 36 scalar/tensor form factors `F1..F36` directly as
     `sum_j c<j>p<i> * AmpT<j>d`, writes `Form_Factors/FF<i>.inc` and
     `../Numerical_Evaluation_1loop/gg1l_results_AG/FF<i>.m`.
   - `LL` — builds 18 helicity form factors `He1..He18` for the
     longitudinal-longitudinal helicity configuration
     (`../Tensor_decomposition/helicity_coeff_LL.inc`), writes
     `Helicity_form_factors/He<i>LL.inc` and `..._results_AG/He<i>_LL.m`.
     Currently the *only* prog_4 variant with the `#include
     helicity_coeff_LL.inc` line un-commented.
   - `LR` — same as LL but for the LR helicity configuration
     (`helicity_coeff_LR.inc`), writes `He<i>LR.inc` /
     `He<i>_LR.m`.
   When adding a new helicity configuration, copy the pattern from
   `prog_4_ggVVLR_1L.frm` (it's the cleanest of the three) rather than `FF`
   or `LL`, which still carry commented-out alternate code paths.

## Directory contents

- `Diagrams/` — per-diagram FORM output from stage 1 (`diagram<N>.inc`,
  N=1..8), each just `id diag<N> = <expression>;`.
- `Tensors_x_amp/` — `AmpT<N>d.inc` (tensor × amplitude, N=1..36) and their
  `Simplified_AmpT<N>d.inc` counterparts (post-simplification versions).
- `Form_Factors/` — final `FF<N>.inc` (N=1..36) scalar form factor output.
- `Helicity_form_factors/` — final `He<N>{LL,LR}.inc` (N=1..18) helicity
  form-factor output.
- `Scalar_products/ScalProd_1L.inc` — rewrite rules mapping loop-momentum
  dot products (`k1.k1`, `k1.p1`, ...) to inverse propagators `D1..D4` per
  topology label (`PL00M`, `PL00Mx12`, `PL0M0`, `PL0M0x12`, ...), used during
  IBP/tensor reduction.
- `Topos1L.rec` — Reduze-notation topology/sector definitions (propagator
  monomial → `D1^n1*D2^n2*...` normalization) per labeled sector.
- `xformxx*.str` — currently empty placeholder files.

## Conventions to preserve when editing `.frm`/`.inc` files

- Symbol/index/vector declarations at the top of each `prog_N` file are
  copy-pasted across files with minor additions per stage; when adding a new
  symbol used by a later stage, add it consistently to that stage's block
  (and any earlier stage whose declarations get echoed forward), following
  the existing `autodeclare` / `cf` / `s` categorization (kinematics,
  couplings, indices, vectors, particle functions, propagator/vertex
  functions, color objects, spinor-helicity functions, etc.).
- Kinematic invariants: `s`, `t`, `u` are Mandelstam-like invariants; `m32`,
  `m42` are the squared masses of `Vboson1`/`Vboson2` (legs `p3`/`p4`); `mt`
  is the internal (quark loop) mass.
- Diagram/topology labels (`d1..d8`, `PL00M`, `PL0M0`, `NPL0M`, and their
  `x12`-suffixed partners under momentum exchange `p1<->p2`) are matched by
  name across `ggVV.1loop`, `Topos1L.rec`, and `Scalar_products/*.inc` —
  keep names consistent if adding a new topology.
- `#write` statements at the end of each stage are the authoritative list of
  what that stage produces; when changing what a stage computes, update the
  matching `#write` loop bounds (e.g. the `#do i=1,36` / `#do i=1,18`
  ranges) to match.
